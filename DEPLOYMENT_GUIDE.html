2

Firebase Project Setup

2.1 Create a New Firebase Project

Go to console.firebase.google.com → Click "Add project".
Name the project dhanvantari-hospital (or your preferred name) → Continue.
Disable Google Analytics (not required) → Click "Create project" → Wait for setup → Click Continue.

2.2 Enable Authentication

In the Firebase Console left sidebar: Build → Authentication → Click "Get started".
Under the "Sign-in method" tab → Click Email/Password → Enable the first toggle → Click Save.
This is used for all staff logins (admin, receptionist, pharmacist, doctor).
Click Google provider → Enable → Set the support email → Click Save.
This is used for the patient portal (/portal) — patients log in with Google.

2.3 Create Firestore Database

Sidebar: Build → Firestore Database → Click "Create database".
Choose "Start in production mode" — you will deploy proper security rules in the next step. Click Next.
Select location: asia-south1 (Mumbai) for India. Click Enable. Wait for provisioning.

2.4 Deploy Firestore Security Rules

The project includes pre-written Firestore security rules. Deploy them from the project folder:

# Install Firebase CLI if not already installed npm install -g firebase-tools # Log in to Firebase firebase login # Navigate to the project folder cd "C:\Users\z004n5ra\Desktop\SUB Projects\MYW\TemplateS\Clients\ABHospital" # Deploy Firestore rules only firebase deploy --only firestore:rules # Deploy Storage rules only firebase deploy --only storage
If firebase deploy asks to select a project, run firebase use --add first and select your Firebase project.

2.5 Get Firebase Client Config Keys

In Firebase Console: click the gear icon (Settings) top-left → Project settings.
Scroll to "Your apps" → Click the Web icon (</>) → Register the app as dhanvantari-hospital-web → Continue.
Copy the values from the firebaseConfig object shown — you will need these in the next section.

First Admin User Setup

No scripts, no service account key, no Firebase Admin SDK. Just two steps in the Firebase Console.

How role management works: Every staff user's role is stored as a Firestore document at /userRoles/{uid} with a single field role: "admin" (or receptionist, pharmacist, doctor). The app reads this document on login and redirects the user to their panel.

Step 1 — Create the admin user in Firebase Auth

Firebase Console → Authentication → Users → Click "Add user".
Email: admin@dhanvantarihospital.in, Password: a strong password → Click Add user.
The new user appears in the list. Copy the User UID (long alphanumeric string in the UID column — e.g., aBc123xyz...).

Step 2 — Set the admin role in Firestore

Firebase Console → Firestore Database → Click "Start collection".
Collection ID: userRoles → Click Next.
Document ID: paste the UID you copied from Authentication (do not use "Auto-ID").
Add field: Name = role, Type = string, Value = admin. Click Save.
# Firestore document structure for the admin user: Collection: userRoles Document ID: {firebase_auth_uid} ← paste the UID here role: "admin" email: "admin@dhanvantarihospital.in"

Step 3 — Test the admin login

Go to http://localhost:3000/login (or your production URL once deployed).
Log in with the admin email and password.
Should redirect to /admin with the full admin sidebar visible.

Step 4 — Create other staff accounts (from admin panel)

Once you have admin access, create all other staff accounts directly from the system — no Firebase Console needed:

Login as admin → Go to /admin/roles.
Fill in: Email, Password, Display Name, Role (Receptionist / Pharmacist / Doctor) → Click "Create Account".
The account is created in Firebase Auth and the Firestore userRoles document is written automatically.
After this step, staff can log in at /login and be redirected to their role-specific panel immediately.

Deploy to Vercel

6.1 Push code to GitHub

# Inside the project folder, initialise git if not done git init git add . git commit -m "Initial commit — Dhanvantari Hospital system" # Create a new private GitHub repository (via github.com) # Then push to it: git remote add origin https://github.com/your-username/dhanvantari-hospital.git git push -u origin main
Make sure .env.local is listed in .gitignore before pushing. Never commit environment variables to a public repository.

6.2 Import into Vercel

Go to vercel.com → Sign in → Click "Add New Project".
Click "Import Git Repository" → Connect your GitHub account if not already connected → Select the dhanvantari-hospital repository.
Vercel auto-detects the framework as Next.js. Leave all build settings at their defaults.
Before clicking Deploy — expand the "Environment Variables" section. Add every variable from your .env.local file one by one. This is critical — the deployment will fail without Firebase keys.
Set NEXT_PUBLIC_SITE_URL to your intended production URL (e.g., https://www.dhanvantarihospital.in) — even before the domain is configured.
Click "Deploy". Vercel builds the project (usually 2–4 minutes). Watch the build logs for any errors.
On success, Vercel assigns a URL like dhanvantari-hospital.vercel.app. Open it and verify the homepage loads.

6.3 Add Firebase Authorized Domain

Firebase Auth requires your Vercel domain to be whitelisted before login will work:

Firebase Console → Authentication → Settings → Authorized domains.
Click "Add domain" → Add dhanvantari-hospital.vercel.app.
After custom domain is configured (Section 8), add the custom domain as well: e.g., www.dhanvantarihospital.in.
Without this step, Firebase Auth will throw an "auth/unauthorized-domain" error when staff try to log in from the Vercel URL.

Post-Deploy Verification

After the first successful deployment, run through these checks before handing over to the client:

Open the deployed URL → verify the homepage loads with the correct hospital name (not a placeholder).
Go to /login → log in as admin → confirm redirect to /admin.
Load demo data from /admin/settings → verify all 4 role logins work (admin, receptionist, pharmacist, doctor).
Issue a queue token → call next → verify TV display at /queue/display updates.
Create an in-patient card (IPD and OPD) → verify print layout.
Create a hospital invoice → print it → verify it opens correctly.
Submit a test appointment from /appointments → verify it appears in /desk/enquiries.
Test patient portal at /portal/login → log in with Google → verify profile page.
Download a backup from /admin/settings → verify the JSON file contains all collections.
Clear demo data (Reset Data) before the client starts using the system with real data.

Custom Domain Setup

8.1 Add domain in Vercel

Vercel Dashboard → Select your project → Settings → Domains.
Enter your domain (e.g., dhanvantarihospital.in) → Click Add. Add both the apex domain and www subdomain.
Vercel will show you the DNS records to add. Copy them — you need to add these at your domain registrar.

8.2 Configure DNS at your registrar

Log in to wherever you registered the domain (GoDaddy, Bigrock, Hostinger, etc.) and add these DNS records:

TypeName / HostValue / Points ToTTL
A @ (apex domain) 76.76.21.21 (Vercel IP) Auto / 3600
CNAME www cname.vercel-dns.com Auto / 3600
DNS propagation can take 10 minutes to 48 hours. Once active, Vercel automatically provisions and renews the SSL certificate. No manual SSL configuration needed.

8.3 Add domain to Firebase Auth authorized domains

Firebase Console → Authentication → Settings → Authorized domains → Click "Add domain".
Add both: dhanvantarihospital.in and www.dhanvantarihospital.in.

8.4 Update NEXT_PUBLIC_SITE_URL in Vercel

Vercel Dashboard → Project → Settings → Environment Variables.
Update NEXT_PUBLIC_SITE_URL to your final production URL: https://www.dhanvantarihospital.in.
Trigger a redeploy: Vercel Dashboard → Deployments → Redeploy (latest deployment).

WhatsApp Setup via Twilio (Optional)

Skip this section for initial go-live. Add WhatsApp when the client is ready to activate automated patient notifications.

9.1 Create Twilio Account & Get Credentials

Sign up at twilio.com → Verify phone number → Go to Console.
Copy Account SID and Auth Token from the Console Dashboard.
Add both to Vercel environment variables and to your .env.local: TWILIO_ACCOUNT_SID and TWILIO_AUTH_TOKEN.

9.2 Test with WhatsApp Sandbox (Development)

Twilio Console → Messaging → Try it out → Send a WhatsApp message.
Follow instructions to join the sandbox: send join [your-code] from your phone to +14155238886.
Set TWILIO_WHATSAPP_FROM=whatsapp:+14155238886 in your env vars.
Issue a token from /desk/queue → click Call Next → verify WhatsApp message received.

9.3 Production WhatsApp Business Setup

Apply for a Twilio WhatsApp Business Sender at Twilio Console → Messaging → WhatsApp senders.
Provide: business name (Dhanvantari Hospital), Facebook Business Manager ID, website URL.
Twilio submits the approval request to Meta. Approval typically takes 1–5 business days.
On approval, update TWILIO_WHATSAPP_FROM with the approved WhatsApp number: whatsapp:+91XXXXXXXXXX.

9.4 Configure Webhook for WhatsApp Bot

In Twilio Console → Messaging → Senders → WhatsApp → your sender → Find "Webhook URL" field.
Set the webhook URL to: https://www.dhanvantarihospital.in/api/whatsapp/webhook
Test by sending "HELP" or "APPOINTMENT" to the WhatsApp number. Bot should auto-reply.
For local webhook testing, use ngrok: run ngrok http 3000 and use the HTTPS URL as your Twilio webhook (e.g., https://abc123.ngrok.io/api/whatsapp/webhook).

All application routes and who can access them:

Full-screen TV display — no header/footer, voice announcements

Public — No login

Patient review submission — standalone page, no website nav

Patients (Google login)
/portal

Patient portal — appointments, records, profile (Google Sign-In)

Staff (Email login)
/login

Single staff login page — redirects by role on success

Admin only
/admin

Admin dashboard, analytics, blog, gallery, reviews, roles, settings

Receptionist + Admin
/desk

Desk panel — queue, cards, billing, enquiries, appointments, wishes

Doctor + Admin
/doctor

Doctor queue view — call next, serve, done, skip. Read-only records.

Pharmacist + Admin
/admin/pharmacy

Pharmacy ERP — medicines, stock, billing, suppliers, reports

Role redirect behaviour: If a user navigates to a route outside their role, the app redirects them to their own panel (e.g., receptionist trying /admin is sent to /desk/queue). Firestore security rules enforce this at the data layer as well — not just the UI.

  • Firestore database created in asia-south1 (Mumbai)
  • Firestore security rules deployed from firestore.rules
  • Firebase Storage enabled and storage rules deployed from storage.rules
  • Firebase web app registered and config keys copied
  • Environment Variables
  • All NEXT_PUBLIC_FIREBASE_* variables filled in .env.local
  • All hospital branding variables set (name, phone, address, email, WhatsApp)
  • RESEND_API_KEY obtained from resend.com and added
  • NEXT_PUBLIC_SITE_URL set to production domain URL
  • REVALIDATION_SECRET set to a unique random string
  • All env vars added to Vercel project settings (not just .env.local)
  • .env.local is NOT committed to Git (in .gitignore)
  • First Admin User
  • Admin user created in Firebase Auth → UID copied
  • Firestore userRoles/{uid} document created with role: "admin"
  • Admin login tested — redirects to /admin with full sidebar
  • All other staff accounts created from /admin/roles
  • All 4 role logins tested and redirected correctly
  • Vercel Deployment
  • Code pushed to private GitHub repository
  • Repository imported into Vercel and deployed
  • Build completed successfully (no build errors in Vercel logs)
  • Vercel deployment URL tested — homepage loads correctly
  • Vercel URL added to Firebase Auth authorized domains
  • Custom Domain
  • Domain registered (e.g., dhanvantarihospital.in)
  • A record and CNAME added at domain registrar pointing to Vercel
  • Domain added in Vercel project → SSL certificate issued automatically
  • Custom domain added to Firebase Auth authorized domains
  • NEXT_PUBLIC_SITE_URL updated to production domain in Vercel env vars
  • Redeployment triggered after env var update
  • Content & Branding
  • Hospital logo at /public/images/logo.jpg
  • Emergency phone number correct on header banner
  • Doctor profiles filled with real names, credentials, specialties
  • Billing quick-add services configured for hospital's actual charges
  • Medicine catalog populated with core medicines
  • Post-Launch
  • Demo data loaded, all modules tested end-to-end
  • Demo data cleared before client starts using the system
  • First real backup downloaded and stored safely
  • Staff training completed — admin, receptionist, pharmacist panels
  • Client has credentials for all staff accounts
  • Twilio WhatsApp configured (if requested)
